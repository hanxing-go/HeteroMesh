## 一、做了什么

删除了旧的单模块结构（`pom.xml` + `src/`），新建了三模块 Maven 项目：

```
HeteroMesh/                              ← Git 仓库根目录
│
├── pom.xml                              ← 父 POM
├── .gitignore
├── Readme.md
├── Plan/
│
├── heteromesh-common/                   ← 子模块 1：公共库
│   ├── pom.xml
│   └── src/main/
│       ├── java/com/heteromesh/
│       │   ├── protocol/                （待写：协议定义）
│       │   └── transport/               （待写：网络传输层）
│       ├── proto/message.proto          Protobuf 消息定义
│       └── resources/logback.xml        日志配置
│
├── heteromesh-controller/               ← 子模块 2：调度节点
│   ├── pom.xml
│   └── src/main/java/com/heteromesh/controller/
│
└── heteromesh-worker/                   ← 子模块 3：计算节点
    ├── pom.xml
    └── src/main/java/com/heteromesh/worker/
```

## 二、每个文件的作用

### 2.1 父 POM — `HeteroMesh/pom.xml`

这是整个项目的「总控文件」，**只做两件事**：

**第一件事：声明有哪些子模块**
```xml
<modules>
    <module>heteromesh-common</module>
    <module>heteromesh-controller</module>
    <module>heteromesh-worker</module>
</modules>
```
这意味着你在项目根目录执行 `mvn compile`，Maven 会**按依赖顺序**编译所有三个模块。

**第二件事：统一管理所有依赖的版本**
```xml
<properties>
    <netty.version>4.1.119.Final</netty.version>
    <protobuf.version>4.30.2</protobuf.version>
    ...
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>io.netty</groupId>
            <artifactId>netty-all</artifactId>
            <version>${netty.version}</version>   ← 版本号在这里集中定义
        </dependency>
        ...
    </dependencies>
</dependencyManagement>
```

为什么这样做？因为 `heteromesh-common` 和 `heteromesh-worker` 都可能引用 Netty。如果版本号散落在各子模块的 pom.xml 里，将来升级要改 N 个地方。集中管理后，改一处全局生效。

**注意：** `<dependencyManagement>` 只是**声明**版本，并不实际引入依赖。子模块需要在 `<dependencies>` 里声明「我要用 Netty」，但不需要写版本号，版本号从父 POM 继承。

---

### 2.2 `heteromesh-common/pom.xml` — 公共模块

```xml
<parent>
    <groupId>com.heteromesh</groupId>
    <artifactId>HeteroMesh</artifactId>
    <version>1.0-SNAPSHOT</version>
</parent>

<artifactId>heteromesh-common</artifactId>
<packaging>jar</packaging>
```

关键点：
- `<parent>` 指向父 POM，继承版本号
- `<packaging>jar</packaging>` 表示编译产出是 `.jar` 文件，可以给其他模块引用
- 这个模块包含了 **Netty、Protobuf、Guava、Gson、Lombok、SLF4J/Logback** 的依赖

**为什么这些依赖放在 common 里？** 因为协议编解码（Netty + Protobuf + Gson）和传输层是 Controller 和 Worker 都要用的。放在 common 里，两者引用 common 就自动获得这些依赖。

---

### 2.3 `heteromesh-controller/pom.xml` 和 `heteromesh-worker/pom.xml`

```xml
<dependencies>
    <dependency>
        <groupId>com.heteromesh</groupId>
        <artifactId>heteromesh-common</artifactId>
        <version>${project.version}</version>   ← 引用同项目的 common 模块
    </dependency>
</dependencies>
```

它们**只依赖 common**，不自己引入 Netty 等库。通过引用 common，间接获得所有需要的依赖（这叫「传递依赖」）。

### 2.4 `message.proto` — Protobuf 消息定义

路径：`heteromesh-common/src/main/proto/message.proto`

```protobuf
syntax = "proto3";
package com.heteromesh.protocol;

// 消息类型枚举
enum ProtoMessageType {
  PING = 0;            // 心跳请求
  PONG = 1;            // 心跳响应
  TASK_REQUEST = 16;   // 推理任务请求
  TASK_RESPONSE = 17;  // 推理任务响应
  REGISTER = 32;       // 节点注册
  REGISTER_ACK = 33;   // 注册确认
  HEARTBEAT = 48;      // 心跳包（区别于 PING/PONG 的应用层心跳）
  HEARTBEAT_ACK = 49;  // 心跳确认
}

// 统一消息体
message ProtoMessage {
  ProtoMessageType type = 1;   // 消息类型
  string request_id = 2;       // RPC 请求ID，用于匹配请求和响应
  string body = 3;             // JSON 字符串，携带具体业务数据
}
```

**Protobuf 是什么？** Google 开发的一种二进制序列化格式。你写 `.proto` 定义文件，然后用 `protoc` 编译器自动生成 Java 类。相比 JSON：
- 体积小约 70%（数字用变长编码，字段名不传输，只传字段编号）
- 反序列化快约 3-5 倍（不需要解析字符串，直接读二进制）
- 向后兼容（加新字段不影响旧版本）

**等号后面的数字不是值，是字段编号**。`type = 1` 意思是 type 字段在二进制流里的编号是 1。Protobuf 用编号而不是字段名来识别字段，这样改了字段名也不影响兼容性。

**消息类型的值为什么有间隔？** `PING=0, PONG=1, TASK_REQUEST=16, REGISTER=32, HEARTBEAT=48`——留间隔是为了以后可以在中间插入新类型，比如 `TASK_CANCEL = 18`。

### 2.5 `logback.xml` — 日志配置

路径：`heteromesh-common/src/main/resources/logback.xml`

```xml
<configuration>
    <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <logger name="com.heteromesh" level="DEBUG"/>   ← 你自己的代码打 DEBUG 级别日志
    <logger name="io.netty" level="INFO"/>           ← Netty 日志只打 INFO 以上
    <root level="INFO"/>
</configuration>
```

- `ConsoleAppender`：日志打到控制台（开发阶段方便）
- `com.heteromesh` 设为 DEBUG：你写的代码可以打详细日志
- `io.netty` 设为 INFO：Netty 内部日志太多了，只显示重要信息
- `%d{HH:mm:ss.SSS} [%thread] %-5level`：日志格式 = 时间 + 线程名 + 级别 + 内容

## 三、为什么这样设计

### 3.1 多模块 vs 单模块

| 维度 | 单模块 | 多模块 |
|------|--------|--------|
| **编译速度** | 改一处，全量编译 | 只编译变更模块 |
| **依赖隔离** | 所有代码互相可见，容易写出循环依赖 | Controller 不能直接 import Worker 的类 |
| **部署** | 打一个 fat jar，Controller 带着 Worker 的代码 | 各打各的 jar，体积小，部署灵活 |
| **面试印象** | "这是一个练习项目" | "这是一个正式项目" |

### 3.2 为什么 common 是独立的

```
                    ┌──────────────┐
                    │   common     │
                    │  协议+传输    │
                    └──────┬───────┘
                           │ 依赖
              ┌────────────┼────────────┐
              ▼            │            ▼
     ┌────────────┐        │   ┌────────────┐
     │ controller │        │   │   worker   │
     │  调度逻辑   │        │   │  推理逻辑   │
     └────────────┘        │   └────────────┘
              │            │            │
              └────────────┼────────────┘
                    不能互相引用
```

Controller 和 Worker 不能直接 import 对方的类——这保证了它们可以在不同 JVM 甚至不同机器上运行。所有共享的东西（消息格式、编解码器）都在 common 里。

### 3.3 Worker 主动出站连接的设计

传统 RPC 模型：Server 监听端口，Client 连接 Server。

我们的模型：
```
云服务器 (公网IP)                  5090 笔记本 (NAT 后)
┌──────────────┐                 ┌──────────────┐
│  Controller  │  ◄── TCP 连接 ──│   Worker     │
│  ServerBootstrap               │   Bootstrap  │
│  监听 :9090   │                 │  出站连接     │
└──────────────┘                 └──────────────┘
```

Worker **主动出站连接** Controller，而不是 Controller 连 Worker。原因：
- 5090 笔记本在家里的路由器后面（NAT），没有公网 IP
- 云服务器有公网 IP
- 内网设备可以主动连公网，反过来不行

这和 [Hive](https://github.com/VakeDomen/HiveCore) 项目的设计一致，也是现实中分布式系统的常见做法。

## 四、依赖版本选择理由

| 依赖 | 版本 | 理由 |
|------|------|------|
| Netty | 4.1.119.Final | 最新的 4.1.x 稳定版，修复了旧版安全漏洞 |
| Protobuf | 4.30.2 | 最新稳定版，支持 Java 21 |
| Guava | 33.4.8-jre | `-jre` 后缀是官方推荐的纯 JRE 版本，不受 Android 限制 |
| Gson | 2.10.1 | 稳定够用，不需要更重的 Jackson |
| Lombok | 1.18.38 | `scope=provided` 表示只在编译时需要，不打包进 jar |
| Logback | 1.5.16 | SLF4J 的实际实现 |
| JMH | 1.37 | OpenJDK 官方基准测试工具 |
