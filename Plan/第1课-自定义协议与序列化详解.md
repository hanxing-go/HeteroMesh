## 一、这节课解决了什么问题

两台电脑通过网络通信，网络只能传字节（010101...）。你需要一个规则：

```
发送方：把「帮我跑 AI」这句话 + 元信息 → 编码成一串字节 → 发出去
接收方：收到一串字节 → 按规则解析 → 还原成「帮我跑 AI」+ 元信息
```

这节课写了 3 个文件，分别解决其中的一环：

| 文件 | 解决的现实问题 |
|------|---------------|
| `MessageType.java` | 定义消息标签：PING/PONG/TASK_REQUEST/TASK_RESPONSE |
| `Message.java` | 装数据的信封：标签 + 请求编号 + 内容 |
| `MessageSerializer.java` | 打包/拆包机器：信封 ↔ 字节 互转 |

## 二、自定义二进制协议格式

```
每条消息的字节布局：
┌──────────────┬────────┬────────┬──────────────┬───────────────┐
│  魔数         │ 版本   │ 类型   │ 消息体长度    │  消息体          │
│  4 字节       │ 1 字节 │ 1 字节 │ 4 字节       │  变长 UTF-8      │
│  0xCAFEBABE  │ 0x01   │ 见枚举 │ 大端序 int   │  JSON 信封       │
└──────────────┴────────┴────────┴──────────────┴─────────----──┘
│←─────────── 固定 10 字节头 ──────────→│←── bodyBytes.length ──→│
```

**各字段说明：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| 魔数 | 4 | `0xCAFEBABE`，Java class 文件的同款魔数。收到数据先校验，不匹配就断开（说明不是本协议） |
| 版本 | 1 | `0x01`，将来协议升级时区分新旧版本 |
| 类型 | 1 | PING=0x01, PONG=0x02, TASK_REQUEST=0x10, TASK_RESPONSE=0x11 |
| 消息体长度 | 4 | 大端序 int，表示消息体有多少字节（不含头） |
| 消息体 | 变长 | JSON 字符串的 UTF-8 字节 |

**为什么消息体用 JSON？** 方便调试——打开抓包工具可以直接看到内容，不用二进制解析。

**为什么还要有 Protobuf？** JSON 体积大、解析慢。Protobuf 体积约 JSON 的 1/3，解析快 3-5 倍。第 5 课用 JMH 实测对比。

## 三、消息体是一个「JSON 信封」

消息体不只是业务数据，还包装了一层：

```json
{
  "requestId": "abc-123-def-456",
  "payload": "什么是虚拟线程？"
}
```

**为什么要包一层？** 协议头只有 10 字节，放不下 36 位的 UUID。requestId 必须跟着 body 一起走——这个设计叫「信封模式」。

```
Message 对象                           线路上传输的 body（JSON）
┌──────────────────────┐   encode      ┌──────────────────────────┐
│ type: TASK_REQUEST   │ ───────────→  │ {"requestId":"abc-123", │
│ requestId: "abc-123" │               │  "payload":"什么是虚拟线程"}│
│ body: "什么是虚拟线程"  │ ←───────────  └──────────────────────────┘
└──────────────────────┘   decode
```

## 四、逐文件详解

### 4.1 MessageType.java — 消息标签

```java
public enum MessageType {
    PING((byte) 0x01),          // 心跳请求：「你还在吗？」
    PONG((byte) 0x02),          // 心跳应答：「我还在」
    TASK_REQUEST((byte) 0x10),  // 推理任务请求：「帮我跑这个 prompt」
    TASK_RESPONSE((byte) 0x11); // 推理任务应答：「这是结果」
    
    private final byte code;
    
    // 枚举构造函数必须是 private
    MessageType(byte code) { this.code = code; }
    
    public byte getCode() { return code; }
    
    // 核心方法：收到一个字节 → 找回对应的枚举值
    public static MessageType fromCode(byte code) {
        for (MessageType type : values()) {  // values() 由 Java 自动生成
            if (type.code == code) return type;
        }
        throw new IllegalArgumentException("未知的消息类型: " + code);
    }
}
```

**关键知识点：**
- `values()` 是 Java 编译器自动生成的，返回所有枚举值的数组
- `fromCode()` 遍历所有枚举值，找 code 匹配的
- 消息类型编号留了间隔（PING=0x01, PONG=0x02, TASK_REQUEST=0x10），方便以后插入新类型

### 4.2 Message.java — 消息实体

```java
public class Message {
    private MessageType type;    // 消息类型
    private String requestId;    // RPC 请求编号（UUID）
    private String body;         // 业务负载
    
    // 全参构造
    public Message(MessageType type, String requestId, String body) { ... }
    
    // 静态工厂方法 —— 简化创建
    public static Message createPing() { ... }
    public static Message createPong() { ... }
    public static Message createTaskRequest(String body) { ... }
    public static Message createTaskResponse(String requestId, String body) { ... }
}
```

**设计选择：为什么 version 和 length 不在 Message 里？**

| 字段 | 在 Message 里？ | 理由 |
|------|:---:|------|
| type | ✅ | 业务需要知道这是什么消息 |
| requestId | ✅ | RPC 请求-响应匹配 |
| body | ✅ | 业务数据 |
| version | ❌ | 传输层概念，只在序列化时写入/读取 |
| length | ❌ | 不是业务数据，是**计算**出来的（body 多长就填多少） |

**静态工厂方法有什么用？** 封装了 `new` 的细节：

```java
// 没有工厂方法：每次都要手动生成 UUID，传空字符串
Message ping = new Message(MessageType.PING, UUID.randomUUID().toString(), "");

// 有工厂方法：一行搞定
Message ping = Message.createPing();
```

**UUID 的作用：** 第 4 课 RPC 会用到。Worker 发请求后，把 requestId 存在 Map 里等回复；Controller 回传时带上 requestId，Worker 才知道「这个回复对应哪个请求」。

### 4.3 MessageSerializer.java — 打包/拆包

这是本节课**最核心、最难写**的类。

#### encode：Message → ByteBuffer

```java
public static ByteBuffer encode(Message msg) {
    // 1. 打包 JSON 信封
    JsonObject envelope = new JsonObject();
    envelope.addProperty("requestId", msg.getRequestId());
    envelope.addProperty("payload", msg.getBody());
    String envelopeJson = gson.toJson(envelope);
    
    // 2. 转 UTF-8 字节（不能用 json.length()，中文一个字符占 3 字节！）
    byte[] bodyBytes = envelopeJson.getBytes(StandardCharsets.UTF_8);
    
    // 3. 分配空间 = 10 字节头 + 消息体字节数
    ByteBuffer buffer = ByteBuffer.allocate(HEADER_LENGTH + bodyBytes.length);
    
    // 4. 按协议格式写入
    buffer.putInt(MAGIC_NUMBER);     // 4 字节
    buffer.put(VERSION);             // 1 字节
    buffer.put(msg.getType().getCode()); // 1 字节
    buffer.putInt(bodyBytes.length); // 4 字节（必须写长度！接收方才知道读多少）
    buffer.put(bodyBytes);           // 剩余字节
    
    // 5. 翻转（写模式 → 读模式）
    buffer.flip();
    return buffer;
}
```

**最容易犯的错：**
- 用 `json.length()` 算大小 → 这是字符数，不是 UTF-8 字节数。中文 1 字符 = 3 字节
- 忘记 `putInt(bodyBytes.length)` → 接收方不知道读多少字节
- 忘记 `flip()` → position 在末尾，接收方读不到数据

#### decode：ByteBuffer → Message

```java
public static Message decode(ByteBuffer buffer) {
    int magic = buffer.getInt();           // 1. 读魔数
    if (magic != MAGIC_NUMBER) throw ...   //    不匹配就报错
    
    byte version = buffer.get();           // 2. 读版本
    byte typeCode = buffer.get();          // 3. 读类型编号
    MessageType type = MessageType.fromCode(typeCode);
    
    int bodyLength = buffer.getInt();      // 4. 读消息体长度
    
    byte[] bodyBytes = new byte[bodyLength]; // 5. 读消息体
    buffer.get(bodyBytes);
    String json = new String(bodyBytes, StandardCharsets.UTF_8);
    
    // 6. 拆 JSON 信封
    JsonObject envelope = gson.fromJson(json, JsonObject.class);
    String requestId = envelope.get("requestId").getAsString();
    String payload = envelope.get("payload").getAsString();
    
    return new Message(type, requestId, payload);
}
```

**读取顺序必须和写入顺序一模一样：** putInt → put → put → putInt → put，对应 getInt → get → get → getInt → get。错一个字节，后面全乱。

## 五、ByteBuffer 快速参考

```java
ByteBuffer buf = ByteBuffer.allocate(10);  // 申请 10 字节

buf.putInt(42);    // 写入 int，占 4 字节，position 后移 4
buf.put((byte)1);  // 写入一个字节，position 后移 1
buf.put(bytes);    // 写入字节数组，position 后移数组长度

buf.flip();        // 切换为读模式（position=0, limit=原position）

int x = buf.getInt();    // 读 int，position 后移 4
byte b = buf.get();      // 读一个字节，position 后移 1
buf.get(byteArray);      // 读一批字节到数组

buf.clear();       // 切换为写模式（position=0, limit=capacity）
```

| 方法 | 作用 | 对 position 的影响 |
|------|------|-------------------|
| `flip()` | 写→读 | position=0, limit=原position |
| `clear()` | 清空，准备重写 | position=0, limit=capacity |
| `rewind()` | 重读 | position=0, limit 不变 |

## 六、学到的错误和经验

| 错误 | 原因 | 正确做法 |
|------|------|---------|
| `allocate(20)` 写死 | 消息体不定长 | 先算总长度再 allocate |
| 用 `json.length()` | 返回字符数而非字节数 | `json.getBytes(StandardCharsets.UTF_8).length` |
| 忘记 `putInt(bodyLength)` | 协议头缺了长度字段 | 必须写入，否则接收方无法切包 |
| `message = null` 后调 setter | 空指针 | 先 `new Message(...)` |
| `String.valueOf(buffer.get(...))` | 语法不匹配 | `new byte[n]` → `buffer.get(arr)` → `new String(arr, UTF_8)` |
| `TASK_RESPONE` | 拼写错误 | `TASK_RESPONSE` |

## 七、单元测试

```java
@Test
void testEncodeDecode() {
    // 1. 准备数据
    Message original = Message.createTaskRequest("什么是虚拟线程？");
    
    // 2. 执行（encode → decode 完整闭环）
    ByteBuffer buffer = MessageSerializer.encode(original);
    Message decoded = MessageSerializer.decode(buffer);
    
    // 3. 验证
    assertEquals(original.getType(), decoded.getType(), "type 应一致");
    assertEquals(original.getRequestId(), decoded.getRequestId(), "requestId 应一致");
    assertEquals(original.getBody(), decoded.getBody(), "body 应一致");
}
```

测试结果：编码 99 字节（10 字节头 + 89 字节 JSON），type/requestId/body 全部解码正确。

## 八、与第 2 课的关系

第 1 课产出的 3 个文件是被第 2 课**调用**的：

```
第 2 课要写的：                          第 1 课产的：
MessageDecoder                            MessageSerializer.decode()
  (Netty 切包) ───────────────────────→   (字节 → Message)

MessageEncoder                            MessageSerializer.encode()
  (Message → 发出去) ──────────────────→   (Message → 字节)
```

第 2 课的编解码器只是「薄薄一层桥接」——Netty 的 ByteBuf ↔ 你的 ByteBuffer。
