# 心跳机制与异常处理详解

> 前置阅读，搞懂长连接为什么需要心跳、IdleStateHandler 怎么用、异常怎么处理

---

## 一、问题场景：从你的第 2 课代码说起

你第 2 课的集成测试跑通了——Client 连上 Server，发消息、收回复。这个连接是**长连接**：

```
Worker (Client) ───────TCP 连接────────→ Controller (Server)
                   建立后一直保持，不会主动断
```

但有个隐藏问题：**TCP 是沉默的**，如果一方悄悄离线了，另一方根本不知道。

```
场景：你的 5090 笔记本（Worker）通过 WiFi 连云服务器（Controller）

  ① 笔记本进入睡眠 / WiFi 断了 / 网线被拔
        ↓
  ② TCP 连接实际已断，但 Controller 端**收不到任何通知**
        ↓
  ③ Controller 还以为 Worker 在线，把任务发过去
        ↓
  ④ 全部超时！用户等半天发现没反应
```

同样的情况：反过来 Controller 挂了，Worker 这边也不知道——还在傻傻等着接收任务。

**心跳就是解决这个问题的**——双方定期互发一条很小的消息，告诉对方「我还活着」。

---

## 二、心跳的类比：保安巡楼

想象你是一个大厦保安，负责巡查每个房间：

```
方式一（不用心跳）：你在前台等着，有人来叫你才知道出事了
方式二（用心跳）：你每隔 30 分钟挨个房间敲门 "有人吗？"
                → 有人回答 → 正常，记录时间
                → 没人回答 → 再敲一次 → 第三次还没人 → 破门进去
```

心跳就是「定期敲门+三次无应答判定离线」这套机制。

## 三、心跳怎么在 Netty 里实现

### 3.1 心跳消息的定义（你已经有了！）

第 1 课你定义 MessageType 时，已经预留了两种心跳消息：

```java
// MessageType.java
PING((byte) 0x01),      // 心跳请求：问 "你还活着吗？"
PONG((byte) 0x02),      // 心跳应答：答 "我还活着！"

// Message.java 里已有工厂方法
Message.createPing();   // type=PING, requestId=UUID, body=""
Message.createPong();   // type=PONG, requestId=UUID, body=""
```

心跳消息的 body 为空，只靠 type 区分。极简开销，每条只占 ~10 字节（协议头）。

### 3.2 IdleStateHandler：Netty 内置的空闲检测器

Netty 提供了一个类叫 `IdleStateHandler`，加到 Pipeline 里就能自动检测「多久没通信了」：

```java
// 构造函数
new IdleStateHandler(
    readerIdleTime,    // 多久没读数据 → 触发 READER_IDLE
    writerIdleTime,    // 多久没写数据 → 触发 WRITER_IDLE
    allIdleTime,       // 多久没读写   → 触发 ALL_IDLE
    timeUnit           // 时间单位
);
```

**它只负责「发现空闲并触发事件」，不负责「怎么处理」。** 处理逻辑写在你的 `HeartbeatHandler` 里。

### 3.3 IdleStateHandler 的工作原理（关键！！！）

```
Pipeline 中的位置很重要：

  [IdleStateHandler] → [MessageDecoder] → [HeartbeatHandler] → [业务Handler]
        ↑                                                      ↑
   必须在最前面！                                     处理心跳事件的地方
```

IdleStateHandler **必须在最前面**，因为它要监控的是**原始字节**层面的读写，而不是解码后的消息。如果你把解码器放在它前面，消息还没解码完就可能被判为空闲了。

它内部做的事：

```
① 初始化：记录当前时间 = T0
② 每次读/写操作发生时，更新时间戳 T_lastRead / T_lastWrite
③ EventLoop 每次循环时检查：
     if (当前时间 - T_lastRead > readerIdleTime)  → 触发 ReaderIdleStateEvent
     if (当前时间 - T_lastWrite > writerIdleTime)  → 触发 WriterIdleStateEvent
     if (当前时间 - max(T_lastRead, T_lastWrite) > allIdleTime) → 触发 AllIdleStateEvent
④ 触发的事件沿 Pipeline 向下传播
⑤ 下一个 Handler 的 userEventTriggered() 方法收到事件
```

### 3.4 我们到底用哪种 Idle？

读空闲、写空闲、全空闲？我们选择 **ALL_IDLE**（全空闲）：

```
为啥不单独用读空闲？
  → 如果对方写了一堆数据过来，我方一直收到，就不是空闲状态
  → 但如果是全空闲——双方都没任何交互——才是真正需要检查的时候

用 ALL_IDLE 最合理：代表「整个连接沉默了」
```

## 四、心跳的完整交互流程

```
Worker (Client)                          Controller (Server)
  |                                            |
  |————— TCP 连接建立 ————————————————————————→|
  |                                            |
  |  IdleStateHandler 定时检查                   |  IdleStateHandler 定时检查
  |                                            |
  |  [30s 无读写] 触发 ALL_IDLE                 |
  |  发 PING ─────────────────────────────────→|  收到 PING
  |                                            |  回复 PONG ─────────────────────────────────→|
  |  收到 PONG，重置计数器                       |
  |                                            |
  |                                            |  [30s 无读写] 触发 ALL_IDLE
  |  收到 PING ←───────────────────────────────|  发 PING
  |  回复 PONG ───────────────────────────────→|
  |                                            |  收到 PONG，重置计数器
  |                                            |
  |  [又过了 30s] 触发 ALL_IDLE                 |
  |  发 PING ─────────(网络断啦!)──────────✗    |
  |  又 30s 还没收到 PONG...                    |
  |  发 PING ─────────(还是发不出去!)──────✗    |
  |  又 30s... 连续 3 次                        |
  |  判定对方离线，关连接！                       |
```

## 五、HeartbeatHandler 要处理的两件事

它会继承 `ChannelInboundHandlerAdapter`，需要重写两个方法：

### 5.1 channelRead → 处理收到的消息

```
收到消息 →
  ├─ type == PING → 回复 PONG
  ├─ type == PONG → 对方还活着，重置计数器
  └─ 其他类型     → 继续传给下一个 Handler（fireChannelRead）
```

注意：如果你收到了 PING 就「吃掉」了（不传给下游），那下游的业务 Handler 就不会看到心跳消息——**心跳是 Handler 之间的私事，不需要污染上层业务。**

同理，收到 PONG 也吃掉，只用来重置计数器。

对于非心跳消息（TASK_REQUEST 等），需要 `ctx.fireChannelRead(msg)` 传给 Pipeline 的下一个 Handler 处理。

### 5.2 userEventTriggered → 处理空闲事件

```
收到事件 →
  ├─ 是 IdleStateEvent 且状态是 ALL_IDLE →
  │    ├─ 计数器 < 3 → 发 PING，计数器+1
  │    └─ 计数器 >= 3 → 对方已死，ctx.close()
  └─ 不是 → 忽略（或传给父类）
```

**关键点：你需要一个 `AtomicInteger` 来记录连续未回复的 PING 次数。**

- 发了一个 PING → 计数器 +1
- 收到 PONG → 计数器归零
- 如果计数器到 3 → 判定离线

为什么用 `AtomicInteger` 而不是普通 `int`？Netty 的 EventLoop 理论上只在一个线程里处理一个 Channel，所以普通 int 其实也安全。但 `AtomicInteger` 更安全而且清晰表达「这个状态可能被多次修改」的意图。

### 5.3 代码骨架（视觉预览）

```java
public class HeartbeatHandler extends ChannelInboundHandlerAdapter {

    private final AtomicInteger missedPongs = new AtomicInteger(0);
    private static final int MAX_MISSED = 3;

    @Override
    public void userEventTriggered(ChannelHandlerContext ctx, Object evt) {
        // 1. 判断是不是空闲事件
        if (evt instanceof IdleStateEvent) {
            IdleStateEvent e = (IdleStateEvent) evt;
            if (e.state() == IdleState.ALL_IDLE) {
                // 2. 连续 3 次就没救了
                if (missedPongs.incrementAndGet() > MAX_MISSED) {
                    ctx.close();
                    return;
                }
                // 3. 否则发 PING
                ctx.writeAndFlush(Message.createPing());
            }
        }
    }

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        Message m = (Message) msg;
        if (m.getType() == MessageType.PING) {
            ctx.writeAndFlush(Message.createPong());
            // 吃掉 PING，不往下传
        } else if (m.getType() == MessageType.PONG) {
            missedPongs.set(0); // 重置计数器
            // 吃掉 PONG，不往下传
        } else {
            ctx.fireChannelRead(msg); // 不是心跳，传给下一个 Handler
        }
    }
}
```

## 六、异常处理 —— ExceptionHandler

### 6.1 Netty 的异常传播机制

Netty 中，异常沿着 Pipeline **反向**传播（从尾向头），直到被某个 Handler 的 `exceptionCaught` 方法捕获：

```
Pipeline: [Decoder] → [Encoder] → [HeartbeatHandler] → [业务Handler]
                                                              ↓
                                                       发生异常！
                                                              ↓
                                          异常向 Pipeline 头部方向传播
                                          依次找每个 Handler 的 exceptionCaught
                                                              ↓
                              ← [ExceptionHandler].exceptionCaught() ← [HeartbeatHandler] ← [业务Handler]
                                      ↑
                              在这里统一处理！不用在每个 Handler 里写重复代码
```

### 6.2 为什么需要 ExceptionHandler

网络编程的异常种类很多：

| 异常 | 含义 | 如何处理 |
|---|---|---|
| `java.io.IOException` | 连接重置、对方断连、网络闪断 | 正常现象，记个日志就行 |
| `io.netty.handler.codec.DecoderException` | 协议解析失败（收到乱码） | 需要打印详情，可能是 bug |
| `java.lang.IllegalArgumentException` | 未知的消息类型 | 代码 bug，需要完整堆栈 |
| 其他异常 | 各种意想不到的错误 | 打印完整堆栈，方便排查 |

如果没有 ExceptionHandler 兜底，异常会直接炸到 EventLoop，Netty 打印一条 WARN 日志然后默默关闭连接——你连发生了什么都不知道。

### 6.3 ExceptionHandler 要做的事

```
收到异常 →
  ├─ IOException → 只打印消息：System.err.println("连接断开: " + ctx.channel())
  └─ 其他异常   → 打印完整堆栈：e.printStackTrace()
最后一步（无论什么异常）：ctx.close() 关闭连接
```

**为什么 IOException 不打印堆栈？** 因为太常见了。用户关浏览器、WiFi 掉线、对方服务器重启——每天可能发生几百次。打堆栈会淹没真正重要的日志。

### 6.4 代码骨架

```java
public class ExceptionHandler extends ChannelInboundHandlerAdapter {

    @Override
    public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
        if (cause instanceof java.io.IOException) {
            System.err.println("[连接断开] " + ctx.channel() + " - " + cause.getMessage());
        } else {
            cause.printStackTrace();
        }
        ctx.close();
    }
}
```

### 6.5 Pipeline 中的最佳位置

ExceptionHandler 应该放在 Pipeline 的**最前面**（或至少很靠前），这样它能兜底所有 Handler 产生的异常：

```java
pipeline.addLast(
    new ExceptionHandler(),      // ← 兜底异常（必须在最前面或很靠前）
    new IdleStateHandler(...),   // ← 检测空闲
    new MessageDecoder(),        // ← 切包+解码
    new MessageEncoder(),        // ← 编码
    new HeartbeatHandler(),      // ← 心跳处理
    new ServerHandler()          // ← 业务处理
);
```

位置有讲究：ExceptionHandler 在 `IdleStateHandler` 和 `MessageDecoder` 之后也能兜底它们的异常，但在 `HeartbeatHandler` 之后也能兜底心跳和业务的异常。**放在最前面最稳**，什么异常都逃不掉。

## 八、Pipeline 处理流程图（一图胜千言）

```
                    ┌──────────────────────────────────┐
Client 发来数据 ──→ │ Pipeline (Inbound 入站方向)        │
                    │                                  │
                    │ ┌──────────────────┐             │
                    │ │ IdleStateHandler │ ← 检测到有数据可读，重置空闲计时器
                    │ └────────┬─────────┘             │
                    │          ↓                        │
                    │ ┌──────────────────┐             │
                    │ │ MessageDecoder   │ ← 字节 → Message 对象
                    │ └────────┬─────────┘             │
                    │          ↓                        │
                    │ ┌──────────────────┐             │
                    │ │HeartbeatHandler  │ ← 判断：心跳？自己处理。业务？往下传
                    │ └────────┬─────────┘             │
                    │          ↓ (仅非心跳消息)         │
                    │ ┌──────────────────┐             │
                    │ │ ServerHandler    │ ← 真正的业务逻辑
                    │ └────────┬─────────┘             │
                    └──────────│──────────────────────┘
                               ↓
                          writeAndFlush(回复)
                               │
                    ┌──────────│──────────────────────┐
                    │ Pipeline (Outbound 出站方向)      │
                    │          ↓                        │
                    │ ┌──────────────────┐             │
                    │ │ MessageEncoder   │ ← Message → 字节
                    │ └────────┬─────────┘             │
                    │          ↓                        │
                    │ ┌──────────────────┐             │
                    │ │ IdleStateHandler │ ← 检测到有数据写出，重置空闲计时器
                    │ └────────┬─────────┘             │
                    └──────────│──────────────────────┘
                               ↓
                           发回 Client

        ★ 如果数据流向是 Outbound（如发送 PING/PONG）：
          Message → Encoder → IdleStateHandler → 网络
        ★ 如果某 Handler 抛异常：
          从当前 Handler 位置沿 Inbound 方向反向传播
          → HeartbeatHandler → IdleStateHandler → ExceptionHandler 捕获
```
