# 心跳机制与异常处理

> 长连接的心跳保活机制 + Netty 异常传播与统一处理

---

## 一、心跳机制

### 为什么需要心跳？

TCP 长连接是「沉默」的——连接断了双方都不知道。典型场景：

```
Worker（你的笔记本）────TCP────→ Controller（云服务器）

笔记本合盖 / WiFi断了 → TCP 连接实际已断
但 Controller 端**收不到任何通知**
还继续把任务发给这个「幽灵」Worker → 全部超时
```

心跳就是**定期互发一条极小的消息**，告诉对方「我还活着」。

### IdleStateHandler：Netty 内置空闲检测器

```java
new IdleStateHandler(
    读空闲时间,   // 多久没读 → READER_IDLE
    写空闲时间,   // 多久没写 → WRITER_IDLE
    全空闲时间,   // 多久没读写 → ALL_IDLE（我们用这个）
    时间单位
);
```

**关键理解：** IdleStateHandler 只负责「发现空闲并触发事件」，不负责「怎么处理事件」。处理逻辑写在你的 HeartbeatHandler 里。

### 心跳交互流程

```
Worker                                Controller
  |                                      |
  |—— 连接建立 ————————————————————————→|
  |                                      |
  |  IdleStateHandler: 30s 无读写         |  IdleStateHandler: 30s 无读写
  |  触发 ALL_IDLE → 发 PING             |  触发 ALL_IDLE → 发 PING
  |                                      |
  |  收到 PING → 回复 PONG → 重置计数器   |  收到 PONG → 重置计数器
  |                                      |
  |  (又过了 30s，一直没人说话)            |
  |  发 PING ───(网络断了!)───→ ✗        |
  |  又 30s... 发 PING → ✗              |
  |  又 30s... 发 PING → ✗              |
  |  连续 3 次，判定离线，close()         |
```

### IdleStateHandler 要放 Pipeline 什么位置？

```
pipeline.addLast(
    ExceptionHandler,          // 最前，兜底一切异常
    IdleStateHandler,          // 第二，监控原始字节的读写
    MessageDecoder,
    MessageEncoder,
    HeartbeatHandler,          // 处理 PING/PONG
    ServerHandler              // 业务
);
```

IdleStateHandler 必须在前面，因为它监控的是**原始字节 Read/Write 操作**。放在 Decoder 后面会导致解码过程耗时算作「非空闲」。

### userEventTriggered vs channelRead

| | `channelRead` | `userEventTriggered` |
|---|---|---|
| 触发时机 | 收到数据（经过解码） | Pipeline 上有人 fire 了事件 |
| 谁触发的 | 对方发数据 | IdleStateHandler 检测到空闲 |
| 参数类型 | `Object msg`（这里是 Message） | `Object evt`（需判断 instanceof） |
| HeartbeatHandler 做什么 | 判断 PING→回PONG / PONG→重置计数 | ALL_IDLE → 发 PING，计数到 3 关连接 |

### 心跳消息的格式

心跳消息直接复用你第 1 课的 Message 协议：

```java
Message.createPing();  // type=PING(0x01), body="", 只有 ~10 字节
Message.createPong();  // type=PONG(0x02), body=""
```

不需要额外的消息格式，body 为空，靠 type 字段区分。

### 计数器设计

```java
private AtomicInteger beatCnt = new AtomicInteger(0);

// 空闲事件：先加再检查
if (beatCnt.incrementAndGet() > MAX_MISSED) {
    ctx.close();  // 判死
    return;
}
ctx.writeAndFlush(Message.createPing());  // 发 PING

// 收到 PING/PONG：归零
beatCnt.set(0);
```

为什么用 `AtomicInteger`？确保操作原子性——虽然单个 EventLoop 是单线程，但 `AtomicInteger` 比 `int` 更安全且清晰表达意图。

## 二、异常处理

### Netty 的异常传播机制

异常沿 Pipeline **反向传播**（从发生位置向 head 方向），依次调用每个 Handler 的 `exceptionCaught()`：

```
Pipeline: [ExceptionHandler] → [IdleState] → [Decoder] → [Handler]
                                                     ↓
                                              这里炸了！
                                                     ↓
                                    异常向 head 方向反向传播
                                                     ↓
                              依次调每个 Handler 的 exceptionCaught()
                                                     ↓
                    → IdleStateHandler.exceptionCaught()
                    → ExceptionHandler.exceptionCaught()  ← 这里统一处理！
```

### 为什么需要 ExceptionHandler？

没有 ExceptionHandler，异常会一直传到 EventLoop，Netty 只打印一条 WARN 日志然后默默关连接——**你连发生了什么都不知道**。

### 异常分类处理

```java
public void exceptionCaught(ChannelHandlerContext ctx, Throwable cause) {
    if (cause instanceof IOException) {
        // 正常断连：对方关了、网络闪断→属于正常现象，只记日志
        System.err.println("连接断开: " + ctx.channel());
    } else {
        // 非 IO 异常：协议解析失败、空指针→代码 bug，打印完整堆栈
        cause.printStackTrace();
    }
    ctx.close();  // 无论什么异常，都要关闭这个出问题的 Channel
}
```

| 异常 | 含义 | 处理方式 |
|---|---|---|
| `IOException` | 连接重置、对方断连、网络闪断 | 正常现象，只打印消息不打印堆栈 |
| `DecoderException` | 协议解析失败（收到乱码） | 可能是 bug，需要完整堆栈 |
| `RuntimeException` | 各种意想不到的错误 | 完整的堆栈帮助排查 |
| `IllegalArgumentException` | 未知消息类型 | 代码 bug |

### IOException 为什么不打印堆栈？

太常见了。用户关浏览器、WiFi 掉线、对方重启——每天几百万次。如果每次打堆栈，日志里全是噪音，真正的 bug 反而被淹没。

### ExceptionHandler 在 Pipeline 的位置

**必须放在 Pipeline 最前面**（head 位置）。因为异常是向 head 方向反向传播的，放在最前面才能兜底所有 Handler 的异常。

## 三、Pipeline 完整布局（第 3 课结束时）

```
Server 端 (HeteroMeshServer):
┌─────────────────────┐
│ ExceptionHandler    │ ← 兜底一切异常
├─────────────────────┤
│ IdleStateHandler    │ ← 30s 无读写 → 触发 ALL_IDLE 事件
├─────────────────────┤
│ MessageDecoder      │ ← 字节 → Message 对象
├─────────────────────┤
│ MessageEncoder      │ ← Message → 字节
├─────────────────────┤
│ HeartbeatHandler    │ ← 收 PING→回 PONG，收 PONG→重置计数，空闲→发 PING
├─────────────────────┤
│ ServerHandler       │ ← 真正的业务逻辑
└─────────────────────┘

Client 端 (WorkerClient): 完全对称，最后是 ClientHandler
```

## 四、面试问答

**Q: 长连接为什么需要心跳？**
A: TCP 是沉默的——连接断了双方感知不到。心跳定期发探测消息，超时无应答判定离线。典型场景：NAT 后的设备（笔记本）连接断了，服务器如果不知道就会一直把任务发给「幽灵」节点。

**Q: IdleStateHandler 怎么工作？**
A: 它记录每次读/写的时间戳，EventLoop 每次循环时检查 `now - lastReadOrWrite > idleTime`，超时就触发 `IdleStateEvent` 沿 Pipeline 传播。它只负责检测，不负责处理——处理逻辑在 `userEventTriggered` 中自定义。

**Q: 为什么 heartbeat 用 ALL_IDLE 而不是 READER_IDLE？**
A: ALL_IDLE 检测的是「整个连接沉默了」。如果对方一直在发数据，就不是空闲状态。用 READER_IDLE 的话，可能对方在发但我们没收到 PONG 就误判。ALL_IDLE 最合理：双方都没任何交互时才需要互相确认存活。

**Q: 3 次心跳无应答才判死，而不是 1 次？**
A: 防止网络抖动。偶尔丢包、路由切换都是暂时的，1 次没收到就断开太敏感了。连续 3 次说明不是偶发，大概率真的离线了。

**Q: Netty 中异常怎么处理的？**
A: Handler 不用主动 try-catch。异常发生后，Netty 自动捕获并按 Pipeline 反向传播（向 head 方向），依次调用各 Handler 的 `exceptionCaught()`。在最前面的 ExceptionHandler 里统一处理：IOException 记日志，其他打堆栈，最后 `ctx.close()`。

**Q: IOException 和其他异常为什么要区别对待？**
A: IOException 是正常现象（对方断连、WiFi 掉线），每天可能发生几百次，打印堆栈会淹没真正的错误日志。其他异常（解析失败、空指针）是代码 bug，需要完整堆栈排查。
