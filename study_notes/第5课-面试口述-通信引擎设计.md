# 面试口述：通信引擎设计（8 问）

---

## 1. 你的通信协议怎么设计的？

**一句话：** 固定长度二进制头（10 字节）+ 可变长度 JSON/Protobuf 消息体。

**协议格式：**

```
┌──────┬───────┬──────┬──────────┬─────────────┐
│ Magic│Version│ Type │ BodyLen  │    Body     │
│ 4B   │  1B   │  1B  │   4B     │  BodyLen B  │
└──────┴───────┴──────┴──────────┴─────────────┘
```

- **Magic（4 字节）**：魔数 `0x484D5347`（"HMSG"），用于快速识别是不是 HeteroMesh 协议包
- **Version（1 字节）**：协议版本号，当前为 `1`
- **Type（1 字节）**：消息类型（PING/PONG/TASK_REQUEST/TASK_RESPONSE/REGISTER）
- **BodyLength（4 字节）**：消息体长度（大端序 int）
- **Body（变长）**：JSON 字符串（开发调试）或 Protobuf 二进制（生产环境）

**为什么这样设计？**
- 固定头 10 字节 → `LengthFieldBasedFrameDecoder` 能精确切分每一帧，解决粘包/半包
- Magic 数字 → 收到垃圾数据时能快速识别并拒绝
- BodyLength 在固定位置（偏移 6，长度 4）→ Netty 的帧解码器直接读取

## 2. 粘包/半包怎么解决？

**粘包：** 发送方连续发了两个小包，TCP 把它们合并成一个 TCP 段发过来 → 接收方读到一个大包

**半包：** 发送方发了一个大包，TCP 拆成多个段 → 接收方第一次只读到前半段

**解决方案：LengthFieldBasedFrameDecoder**

```java
new LengthFieldBasedFrameDecoder(
    10 * 1024 * 1024,  // maxFrameLength = 10MB
    6,                  // lengthFieldOffset = 6（BodyLength 从第 6 字节开始）
    4                   // lengthFieldLength = 4（BodyLength 占 4 字节）
)
```

工作原理：
1. 先读 10 字节固定头
2. 从第 6 字节读 4 字节 → 得到 BodyLength = N
3. 等待 N 字节全部到达（半包时阻塞等待）
4. 返回完整的 10+N 字节帧
5. 多余字节留在缓冲区，留给下一帧（解决粘包）

**关键：** 协议设计时 BodyLength 必须放在固定位置，Netty 才能自动切分。

## 3. 心跳怎么实现？

**为什么需要心跳？**
TCP 连接如果一方异常断开（拔网线、进程 kill），另一方不会立刻知道。可能一直以为连接还活着，发消息石沉大海。

**实现方案：IdleStateHandler + PING/PONG**

```
Pipeline: IdleStateHandler → HeartbeatHandler → BusinessHandler
```

**IdleStateHandler**：Netty 提供的空闲检测器
```java
new IdleStateHandler(0, 0, 10, TimeUnit.SECONDS)  // 10s 无任何读写 → 触发 ALL_IDLE 事件
```

**HeartbeatHandler**：
```java
// 收到 ALL_IDLE 事件 → 发 PING，计数器 +1
userEventTriggered() {
    if (beatCnt.incrementAndGet() > 3) {  // 连续 4 次空闲，发了 3 个 PING 没回
        ctx.close();                       // 判定对方离线，关闭连接
    }
    ctx.writeAndFlush(PING);
}

// 收到 PING → 回复 PONG，计数器归零
channelRead() {
    if (PING) { writeAndFlush(PONG); beatCnt.set(0); }
    if (PONG) { beatCnt.set(0); }         // 收到回复，计数器归零
}
```

**时间线（10s 空闲超时为例）：**
```
0s  → 正常通信
10s → 空闲 → 发 PING #1
20s → 空闲 → 发 PING #2
30s → 空闲 → 发 PING #3
40s → 空闲 → 计数器=4 > 3 → ctx.close()  ← 约 40s 检测到离线
```

两端都装心跳 Handler → 任意一端异常断开，另一端都能检测到。

## 4. RPC 怎么实现？

**一句话：** requestId + CompletableFuture + ConcurrentHashMap 待响应池。

**核心组件：RpcClient**

```java
public class RpcClient {
    // key=requestId, value=等待被填充的 Future
    private ConcurrentHashMap<String, CompletableFuture<Message>> pendingRequests;

    // 发请求 → 返回一个"空盒子"（Future）
    public CompletableFuture<Message> call(String body) {
        Message request = Message.createTaskRequest(body);
        CompletableFuture<Message> future = new CompletableFuture<>();  // 空盒子
        pendingRequests.put(request.getRequestId(), future);            // 登记
        channel.writeAndFlush(request);                                  // 发送
        return future;  // 立刻返回，不阻塞
    }

    // 收到响应 → 匹配 requestId → 填充盒子
    public void onResponse(Message response) {
        CompletableFuture<Message> future = pendingRequests.remove(response.getRequestId());
        if (future != null) {
            future.complete(response);  // 填充！触发 thenAccept 回调
        }
    }
}
```

**完整流程：**
```
Worker                                    Controller
  │                                           │
  │ call("hello")                             │
  │   → 生成 requestId="uuid-1"              │
  │   → 创建 Future①，放入 pendingRequests    │
  │   → 发送 TASK_REQUEST ─────────────────→  │
  │   → return future①（空盒子）              │  处理请求
  │                                           │  生成响应 requestId="uuid-1"
  │   ←──────────────────────────── 回复 ────  │
  │ channelRead0() 收到 TASK_RESPONSE         │
  │   → rpcClient.onResponse(msg)             │
  │   → pendingRequests 中取出 future①        │
  │   → future①.complete(响应)                │
  │   → thenAccept 回调自动执行！              │
```

**关键：** 多请求并发时，响应可能乱序到达。用 requestId 做 key 精确匹配，不会串。

## 5. JSON 和 Protobuf 的区别？你选哪个？为什么？

| | JSON | Protobuf (二进制) |
|---|---|---|
| 可读性 | ✅ 人能直接读 | ❌ 二进制，不可读 |
| 体积 | 大（传输字段名+结构符号） | 小（字段编号+varint） |
| 速度 | 慢（文本解析） | 快（二进制直接映射） |
| 类型安全 | ❌ 弱类型，靠约定 | ✅ .proto 定义，编译期校验 |
| Schema | 无 | .proto 文件 |
| 调试 | curl/postman 直接发 | 需要工具解码 |

**实测数据（100,000 次迭代）：**

| | JSON | Binary | 提升 |
|---|---|---|---|
| 体积 | 126 bytes | 90 bytes | 28.6% |
| 编码 | 1305 ns | 283 ns | 4.6x |
| 解码 | 894 ns | 153 ns | 5.8x |

**我的选择：双方案，通过配置切换。**
- 开发/调试阶段用 JSON，人能看懂，方便排查问题
- 生产环境用 Protobuf（二进制），省带宽、低延迟
- 切换方式：MessageEncoder 中根据配置选择 JSON 还是 Binary Codec

## 6. JMH 是什么？比 System.currentTimeMillis() 好在哪？

**JMH = Java Microbenchmark Harness**，OpenJDK 官方的微基准测试工具。

**为什么不用 System.currentTimeMillis()？**

```java
// 粗糙的计时（不准确）
long start = System.currentTimeMillis();
doSomething();
long end = System.currentTimeMillis();
```

问题：
1. **没有预热**：JVM 刚开始运行是解释执行，跑多了 JIT 编译后才快。第一次跑的结果不代表真实性能
2. **死代码消除**：如果 `doSomething()` 的返回值没人用，JVM 可能直接不执行，测出的时间是 0
3. **没有统计**：只跑一次，不知道波动范围
4. **GC 干扰**：计时期间可能触发 GC，把延迟算到你的代码头上了

**JMH 怎么解决：**
- 预热阶段：先跑几千次不计入结果，让 JIT 生效
- 多轮迭代：跑 N 轮，每轮 M 次，统计 avg/min/max/误差
- Blackhole：防止 JVM 优化掉你的测试代码
- Fork 隔离：每个测试在新 JVM 里跑，互不干扰

**但我们也用了手动计时（ProtocolBenchmark）：** 因为 JMH 跑 Fork 模式时 child JVM 找不到 classpath，改用 `nanoTime() + 预热循环 + 大量迭代` 的方式，原理相同，结果也可靠。

## 7. Worker 突然死掉，Controller 怎么发现？

**靠心跳超时机制。**

```
Worker 正常时：每 10s 收到 PING → 回复 PONG → beatCnt 归零 → 连接保持
Worker 宕机后：
  10s → Controller 空闲 → 发 PING #1 → 无回复（beatCnt=1）
  20s → Controller 空闲 → 发 PING #2 → 无回复（beatCnt=2）
  30s → Controller 空闲 → 发 PING #3 → 无回复（beatCnt=3）
  40s → Controller 空闲 → beatCnt 变成 4 > 3 → ctx.close()
```

ctx.close() 触发后：
1. Channel 关闭 → ConnectionManager 中自动移除
2. 后续一致性哈希路由不再分配任务给这个 Worker
3. 如果有未完成的任务，触发超时重试，路由到其他节点

## 8. 为什么 Worker 主动连 Controller？

**一句话：穿透 NAT。**

你的 5090 笔记本在家里的局域网（192.168.1.x），Controller 在云服务器（公网 IP）。

```
❌ Controller 主动连 Worker：
   Controller(公网) → Worker(内网 192.168.1.x) → 路由器 NAT 挡住了，连不上

✅ Worker 主动连 Controller：
   Worker(内网) → 路由器(NAT) → Controller(公网) → 连接成功！
```

NAT 设备（路由器）允许内网主动发起的出站连接。Worker 主动连 Controller 恰好利用了这个特性。

**类比：** 你在公司内网，可以主动访问百度（公网），但百度的服务器不能主动连接你的电脑。

## Pipeline 全景回顾

```
┌──────────────────────────────────────────────────┐
│               Netty Pipeline（数据流向）           │
│                                                  │
│  入站（读）:                                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │Exception │→│IdleState │→│ Message  │→ ...   │
│  │ Handler  │  │ Handler  │  │ Decoder  │       │
│  └──────────┘  └──────────┘  └──────────┘       │
│       ↑ 最前面     ↑ 空闲检测    ↑ 字节→Message    │
│                                                  │
│  出站（写）:                                      │
│  ... ← Message Encoder ← ...                     │
│         ↑ Message→字节                            │
│                                                  │
│  Pipeline 末尾：HeartbeatHandler → BusinessHandler│
│                                                  │
│  异常传播方向：←←← (从出错位置往 pipeline 头部传)   │
└──────────────────────────────────────────────────┘
```

- **ExceptionHandler 在最前面**：确保任何 Handler 抛出的异常都能被它捕获
- **IdleStateHandler**：检测读写空闲，触发 `userEventTriggered`
- **MessageDecoder**：处理粘包/半包，ByteBuf → Message
- **MessageEncoder**：出站时 Message → ByteBuf，写入协议头
- **HeartbeatHandler**：处理 PING/PONG，传给业务 Handler
- **BusinessHandler**：处理 TASK_REQUEST / TASK_RESPONSE
